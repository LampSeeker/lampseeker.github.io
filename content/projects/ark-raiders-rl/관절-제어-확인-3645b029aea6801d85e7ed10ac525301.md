# 1. 소개


  저번에 이어서 이번에는 다리 관절 하나하나를 제어할 수 있는 구조인지 확인해보도록 하자. 왜냐하면 강화학습을 진행하기 위해서는 이후에 AI 모델이 

- 앞다리 첫 관절을 들어라
- 4 번째 두번째 관절을 30도 접어라 같은

명령을 줄 수 있어야 하기 때문이다.


# 2. 관절 제약(Constraint) 구조 확인


## 2.1. 관절 제약(Constraint) 구조 확인

	- Physics Asset 열기 → 다리 하나를 기준으로 스켈레톤 트리 확인

		![](https://lampseeker.github.io/api?block_id=3645b029-aea6-800e-b2a2-e6c87940741a)


		> 스켈레톤 트리란?  
		>   
		> **스켈레톤의 계층 구조/물리 바디/프리미티브/컨스트레인트 출력 창**

	- 밑의 화면에는 컨스트레인트 그래프(Constraints Graph)도 확인할 수 있는데 이는,

		![](https://lampseeker.github.io/api?block_id=3645b029-aea6-80d7-b2ca-e9fe4d85feb5)


		> **Physics Asset 안에서 “어떤 물리 바디가 어떤 관절 제약으로 연결되어 있는지”를 도식으로 보여주는 창**

		- 노란색 박스 : 컨스트레인트
		- 초록 박스들 : 물리 바디

		로 이루어져 있고, 추후에 관절 하나하나에 모터처럼 목표 각도를 주고, 그걸 강화학습의 Action으로 제어하는 것이 목적이다. 이 그래프를 이용해 나중에

		- 어떤 관절을 움직일 지
		- 다리 1개에 관절이 몇 개 있는지
		- 몸통과 다리가 어떻게 연결되어 있는지

		를 파악한다.


		여기서 최종적으로 제어 대상이 되는 것이 바로 이 노란 컨스트레인트들이다.


## 2.2. 코드로 관절 상세 정보 확인

	- 편집 → 플러그인 → Python Editor Script Plugin 설치

		![](https://lampseeker.github.io/api?block_id=3645b029-aea6-80fb-84d5-ca4a79de851a)

	- 파이썬 코드 작성

		<details>
		  <summary>dump_physics_constraints.py</summary>
		
		
		```python
		import unreal
		
		
		def find_selected_physics_asset():
		    selected_assets = unreal.EditorUtilityLibrary.get_selected_assets()
		
		    for asset in selected_assets:
		        if isinstance(asset, unreal.PhysicsAsset):
		            return asset
		
		    return None
		
		
		def dump_constraints(physics_asset: unreal.PhysicsAsset) -> None:
		    constraints = physics_asset.get_constraints(False)
		
		    unreal.log("=" * 80)
		    unreal.log(f"[PhysicsAsset] {physics_asset.get_name()}")
		    unreal.log(f"[Constraint Count] {len(constraints)}")
		    unreal.log("=" * 80)
		
		    rows = []
		
		    for index, accessor in enumerate(constraints, start=1):
		        _, parent_body, child_body = (
		            unreal.ConstraintInstanceBlueprintLibrary.get_attached_body_names(accessor)
		        )
		
		        parent_name = str(parent_body)
		        child_name = str(child_body)
		
		        rows.append((parent_name, child_name))
		        unreal.log(f"{index:02d}. {parent_name}  <->  {child_name}")
		
		    unreal.log("=" * 80)
		    unreal.log("[Grouped by parent body]")
		
		    grouped = {}
		    for parent_name, child_name in rows:
		        grouped.setdefault(parent_name, []).append(child_name)
		
		    for parent_name in sorted(grouped.keys()):
		        children = ", ".join(sorted(grouped[parent_name]))
		        unreal.log(f"- {parent_name}: {children}")
		
		    unreal.log("=" * 80)
		
		
		def main():
		    physics_asset = find_selected_physics_asset()
		
		    if physics_asset is None:
		        unreal.log_error(
		            "Physics Asset을 콘텐츠 브라우저에서 먼저 선택한 뒤 다시 실행하세요."
		        )
		        return
		
		    dump_constraints(physics_asset)
		
		
		main()
		```

	```text
	FrontLeg1_R  <->  FrontLeg2_R
	```

## 2.4. 컨스트레인트 디테일 패널 분석


	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-8025-8c01-dd5b6d5363ac)

	- 해당 디테일 패널은 선택한 **선택한 컨스트레인트 **`**FrontLeg2_R ↔ FrontLeg1_R**`** 하나의 물리 성질을 전부 조정하는 곳**
	- 쉽게 말하면,

		> **“이 관절이 어디에 있고, 얼마나 움직일 수 있고, 흔들리면 어떻게 보정하고, 모터로 어떻게 움직일지”**  
		> 를 설정하는 패널


	### 상세 정보

	  <summary>1. 조인트 이름 / 자손 본 / 부모 본 이름 ⇒ 부모/자식 스텔레톤</summary>
	
	
	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-80f6-aa6d-fe2ed07d0b69)
	
	
	| 항목          | 의미                                  |
	| ----------- | ----------------------------------- |
	| **조인트 이름**  | 이 컨스트레인트의 이름                        |
	| **자손 본 이름** | 이 관절이 제어하는 아래쪽 본. 지금은 `FrontLeg2_R` |
	| **부모 본 이름** | 기준이 되는 위쪽 본. 지금은 `FrontLeg1_R`      |
	
	
	`**FrontLeg1_R**`**을 기준으로 **`**FrontLeg2_R**`**이 어떻게 움직일지** 정하는 설정

	<details>
	  <summary>2. Constraint Transforms ⇒ 관절 위치/회전축 방향</summary>
	
	
	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-8018-b50c-c5a2ce314c21)
	
	
	**관절의 기준 위치와 기준 회전축**을 정하는 곳
	
	
	| 항목             | 의미                    |
	| -------------- | --------------------- |
	| **자손 위치 / 회전** | 자손 본 쪽에서 본 컨스트레인트 기준점 |
	| **부모 위치 / 회전** | 부모 본 쪽에서 본 컨스트레인트 기준점 |
	
	
	쉽게 비유하면
	
	
	> **“경첩이 정확히 어디에 달려 있고, 어느 축으로 접히는가?”**
	
	
	를 정하는 값
	
	
	예를 들어 무릎 관절이라면, 이 축이 실제 무릎이 접히는 방향과 맞아야 자연스럽게 움직임
	
	
	축이 잘못되어 있으면 **옆으로 꺾이거나 비틀리는 이상한 관절**이 될 수 있음. Physics Asset에서 컨스트레인트의 위치와 회전은 실제 관절 중심과 회전축을 정하는 핵심 설정
	
	
	
	  </details>


	<details>
	  <summary>3. Constraint Behavior ⇒ 관절 자체의 일반적인 동작 방식 정의</summary>
	
	
	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-80f0-92d0-c574ea443084)
	
	
	**관절 자체의 일반적인 동작 방식**을 정하는 영역
	
	
	부모 본과 자손 본이 **서로 충돌할지 말지** 정의
	
	- 체크 ON: 이 관절로 연결된 두 바디끼리는 충돌하지 않음
	- 체크 OFF: 서로 충돌 가능
	
	보통 **연결된 뼈끼리는 충돌을 끄는 경우가 많음**
	
	
	무릎 윗마디와 아랫마디가 붙어 있는데 서로 계속 충돌까지 하면 덜덜 떨 수 있음
	
	
	부모 바디가 자손 바디의 움직임에 **영향을 덜 받게** 하는 옵션
	
	- 체크 OFF: 부모와 자손이 서로 힘을 주고받음
	- 체크 ON: 자손이 흔들려도 부모는 상대적으로 영향을 덜 받음
	
	예를 들어, 몸통 ↔ 다리 관절에서 부모 도미네이트를 켜면, 다리가 흔들려도 몸통이 덜 흔들릴 수 있음
	
	
	다만 너무 남용하면 **물리 반응이 부자연스러워질 수 있음**. 공식 문서도 부모 도미네이트가 켜지면 자식의 운동이 부모에 영향을 주지 않는다고 설명()
	
	
	
	  </details>


	<details>
	  <summary>4. 프로젝션</summary>
	
	
	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-802e-80fe-c2e93ecbc946)
	
	
	**관절이 계산 오차로 벌어지거나 틀어졌을 때 강제로 다시 맞춰주는 보정 기능**
	
	
	| 항목            | 의미                         |
	| ------------- | -------------------------- |
	| **투영 활성화**    | Projection 보정 사용 여부        |
	| **투영 선형 허용치** | 위치 오차가 이 거리보다 커지면 강제로 붙임   |
	| **투영 각도 허용치** | 각도 오차가 이 각도보다 커지면 강제로 바로잡음 |
	| **투영 선형 알파**  | 작은 위치 오차를 얼마나 강하게 보정할지     |
	| **투영 각도 알파**  | 작은 회전 오차를 얼마나 강하게 보정할지     |
	
	
	비유하면:
	
	
	> “관절이 계산상 조금 벌어졌으면, 그냥 두지 말고 다시 붙여라.”
	
	
	라는 안전장치()
	
	
	Projection은 관절을 단단하게 유지하는 데 도움을 주지만, **충돌이 복잡한 체인에서 너무 세게 쓰면 튐이나 지터를 유발할 수 있음.** 공식 문서도 여러 제약이나 충돌과 함께 쓰면 예상치 못한 결과가 날 수 있다고 설명()
	
	
	### 지금은?
	
	
	체크돼 있는 상태 그대로 둬도 돼.

	<details>
	  <summary>5. 쇼크 프로퍼게이션</summary>
	
	
	![](https://lampseeker.github.io/api?block_id=3645b029-aea6-8032-b2e3-f09e8567eea8)
	
	
	이건 긴 관절 사슬이 흐물흐물해지는 걸 줄이기 위한 보정
	
	
	예를 들어 다리가
	
	
	```text
	몸통 → 허벅지 → 종아리 → 발
	```
	
	
	처럼 이어질 때, 가장 끝쪽에서 힘이 걸리면 전체 체인이 불안정하게 흔들릴 수 있음
	
	
	이때 **부모 쪽을 일시적으로 더 무겁게 취급해서 관절 체인을 단단하게** 만드는 옵션이 Shock Propagation
	
	
	| 항목                | 의미          |
	| ----------------- | ----------- |

# 용어 설명


	# 스피어, 트위스트, 스윙


		## 1. 스피어... = SLERP Drive


			UI가 잘려서 `스피어...`로 보이는데, 의미상 **Slerp Drive** 또는 **Spherical Drive** 계열로 보면 돼.


			## 무슨 방식이냐?


			현재 관절 자세에서 목표 자세까지 갈 때,


			> **가장 자연스럽고 짧은 회전 경로로 전체 방향을 맞추려는 방식**


			이야.


			예를 들어 현재 다리 마디가 이 방향이고:


			```text
			현재 회전: 0, 0, 0
			목표 회전: 0, 30, 0
			```


			이면, 스피어 드라이브는


			**“어떤 축 하나만 따로 따지지 말고, 목표 회전 자세 전체로 가자”**고 계산해.


			공식 문서에서는 **SLERP가 shortest spherical path, 즉 가장 짧은 구면 회전 경로를 사용한다**고 설명해.


			## 장점

			- 목표 자세 하나를 주고 움직이기 쉬움
			- 테스트용으로 직관적임
			- 지금처럼 “관절이 목표 각도를 따라 움직이는지” 확인하기 좋음

			---


			## 단점

			- **Angular Constraint 축 중 하나라도 Locked면 SLERP가 작동하지 않음**
			- 나중에 무릎처럼 “한 축만 회전 허용”하는 진짜 경첩 관절을 만들면 SLERP보다 `Twist and Swing` 쪽이 더 맞을 수 있음.

			지금 네 관절은:


			```text
			Swing1 = Limited
			Swing2 = Limited
			Twist = Limited
			```


			즉, **Locked 축이 없어서 SLERP 테스트가 가능한 상태**야.


		---


		## 2. 트위스트 = Twist Drive


			`트위스트`는 이름 그대로:


			> **본의 길이 방향 축을 중심으로 비트는 회전**


			을 제어하는 드라이브야.


			이미지 없이 말로 비유하면:

			- 팔을 앞으로 뻗은 뒤
			- 팔뚝 자체를 시계 방향/반시계 방향으로 **비트는 회전**

			이 `Twist`에 가까워.


			## 로봇 다리에서는?


			관절에 따라 다르지만 보통:

			- 무릎 관절의 주 굽힘에는 별로 필요 없고
			- 다리 루트 관절이나 총구/포탑의 비틀림에는 의미가 있을 수 있어.

			다만 정확히 어떤 회전이 Twist로 잡히는지는


			**Constraint Transform의 축 방향**에 따라 달라져.


		---


		## 3. 스윙 = Swing Drive


			`스윙`은:


			> **축 자체가 옆으로 기울어지거나 휘는 회전**


			을 제어하는 드라이브야.


			비유하면:

			- 어깨에서 팔을 옆으로 들기
			- 팔을 앞뒤로 흔들기

			같은 “기울어지는” 회전이 Swing 계열이야.


			언리얼에서는 Angular Limits도:


			```text
			Swing 1
			Swing 2
			Twist
			```


			로 나뉘지?


			모터의 `스윙`은 그중 **Swing 회전 방향을 구동하는 것**이야. 공식 문서에서도 Twist and Swing 모드는 회전 경로를 Twist와 Swing으로 분해해 제어한다고 설명한다.


		## 정리표


		| 체크박스       | 의미                          | 언제 주로 씀                                |
		| ---------- | --------------------------- | -------------------------------------- |
		| **스피어...** | 목표 회전 자세 전체를 SLERP 방식으로 따라감 | `Angular Drive Mode = SLERP`           |
		| **트위스트**   | 축을 중심으로 비트는 회전 구동           | `Angular Drive Mode = Twist and Swing` |
		| **스윙**     | 축이 기울어지는 회전 구동              | `Angular Drive Mode = Twist and Swing` |

	## 블루프린트 항목 종류

		# 먼저 전체 관계


		가장 중요한 계층만 보면:


		```text
		Actor
		└─ Pawn
		   └─ Character
		```


		그리고 별도로:


		```text
		PlayerController
		GameModeBase
		ActorComponent
		SceneComponent
		```


		가 있어.


		---


		# 1. 액터 Actor


		> **월드에 놓거나 스폰할 수 있는 가장 기본적인 오브젝트**


		예시:

		- 문
		- 총알
		- 상자
		- 적 로봇
		- 우리가 만들 `BP_SpiderRobot`

		즉, **레벨 안에 존재하는 대부분의 일반 오브젝트**는 Actor로 만들 수 있어. Epic 문서도 Actor를 월드에 배치하거나 스폰할 수 있는 객체로 설명한다.


		### 우리 경우


		지금 만드는 거미 로봇은 우선:


		```text
		BP_SpiderRobot → Actor
		```


		로 만들면 돼.


		왜냐하면 지금 단계에서는:

		- 플레이어가 직접 빙의해서 조종하는 캐릭터가 아니고
		- 강화학습 대상 물리 로봇으로 먼저 구성할 거라서

		**Actor가 가장 단순하고 적절해.**


		---


		# 2. 폰 Pawn


		> **플레이어나 AI가 “Possess”, 즉 빙의해서 조종할 수 있는 Actor**


		예시:

		- 자동차
		- 드론
		- 탱크
		- 직접 조작하는 로봇

		Pawn은 Actor의 자식이고, **Controller가 붙어서 조종할 수 있다는 점**이 핵심이야. Epic 문서도 Pawn을 플레이어나 AI가 소유해 제어할 수 있는 Actor의 기본 클래스로 설명한다.


		### 이런 때 Pawn

		- 키보드/패드로 직접 거미 로봇을 조종할 예정
		- AI Controller가 붙어서 명령을 내리게 할 예정
		- 게임 속 유닛처럼 다루고 싶음

		### 우리 프로젝트에서는?


		나중에 **로봇을 직접 조종하거나 AI가 소유하는 형태**로 확장할 때 Pawn을 고려할 수 있어.


		하지만 지금은 **관절 제어 실험용 액터**가 먼저라 Actor로 가는 게 깔끔해.


		---


		# 3. 캐릭터 Character


		> **걷는 인간형 캐릭터를 쉽게 만들기 위해 준비된 Pawn**


		Character는 Pawn의 자식이고, 기본적으로:

		- 캡슐 콜리전
		- 스켈레탈 메시
		- Character Movement Component
		- 걷기, 점프, 중력 이동 처리

		같은 게 이미 붙어 있어. Epic 문서도 Character를 **메시, 충돌, 내장 이동 로직을 가진 Pawn**이라고 설명한다.


		### 예시

		- FPS 플레이어
		- RPG 인간 캐릭터
		- 3인칭 마네킹 캐릭터

		### 우리 거미 로봇에는?


		**거의 안 맞아.**


		왜냐하면 우리는:

		- Character Movement로 걷게 하는 게 아니라
		- **물리 관절과 강화학습으로 걷게 할 것**이기 때문이야.

		그래서 **Character는 선택하지 않는다.**


		---


		# 4. 플레이어 컨트롤러 PlayerController


		> **사람 플레이어의 입력과 시점, 조작 흐름을 담당하는 객체**


		PlayerController는:

		- 키보드/마우스 입력
		- 카메라 회전
		- Pawn 조종
		- HUD나 플레이어 관점 처리

		같은 걸 맡아. Epic 문서도 PlayerController가 인간 플레이어가 Pawn을 제어하는 데 사용된다고 설명한다.


		### 예시

		- WASD 입력 받아 Pawn 이동
		- 마우스로 조준
		- 클릭 입력 처리

		### 지금 만들 건?


		아니야.


		우리는 **로봇 자체**를 만들고 있으니 PlayerController가 아니라 Actor를 골라야 해.


		---


		# 5. 게임 모드 베이스 GameModeBase


		> **게임 규칙을 정의하는 관리자**


		예시:

		- 게임 시작 시 어떤 Pawn을 생성할지
		- 승리/패배 규칙
		- 점수 방식
		- 플레이어 입장 규칙

		GameModeBase는 **“현재 맵에서 어떤 게임을 할 것인가”**를 정하는 규칙 관리자에 가까워. 공식 문서도 게임 규칙, 점수, 허용되는 액터, 입장 조건 등을 관장한다고 설명한다.


		### 우리 프로젝트에서는?


		나중에:

		- 학습용 맵 전용 GameMode
		- 플레이 시작 시 로봇 자동 배치
		- RL 트레이닝 매니저 생성

		같은 걸 정할 때 만들 수 있어.


		하지만 **지금 만드는 거미 로봇 자체는 GameMode가 아니다.**


		---


		# 6. 액터 컴포넌트 Actor Component


		> **액터에 붙여서 기능만 추가하는 재사용 부품**


		Actor Component는:

		- 위치가 없음
		- 보통 눈에 보이지 않음
		- 기능만 담당

		예시:

		- 체력 컴포넌트
		- 인벤토리 컴포넌트
		- 로봇 관절 제어 컴포넌트
		- 강화학습 보상 계산 컴포넌트

		Epic 문서도 ActorComponent를 여러 Actor에 재사용 가능한 행동을 정의하는 기본 컴포넌트로 설명한다.


		### 우리 프로젝트에서는?


		나중에 아주 중요해질 수 있어.


		예를 들어:


		```text
		BP_SpiderRobot
		└─ JointControllerComponent
		```


		이런 식으로


		**관절 제어 로직만 따로 ActorComponent로 분리**할 수 있어.


		하지만 지금 만드는 건 **컴포넌트가 아니라 로봇 본체**이므로 이건 아님.


		---


		# 7. 씬 컴포넌트 Scene Component


		> **위치, 회전, 크기 Transform을 가지는 컴포넌트**


		Scene Component는:

		- 월드 안에서 위치를 가짐
		- 다른 컴포넌트를 자식으로 붙일 수 있음
		- 단독으로는 보통 보이지 않고 충돌도 없음

		공식 문서도 SceneComponent가 Transform과 부착 계층을 가지지만, 자체 렌더링이나 충돌은 없다고 설명한다.


		### 예시

		- 카메라를 붙일 기준점
		- 총구 위치 기준점
		- 미사일 발사 위치
		- 센서 위치

		### 우리 프로젝트에서는?


		나중에 총 발사 위치를 잡을 때 아주 유용해.


		예:


		```text
		BP_SpiderRobot
		└─ MuzzlePoint_L
		└─ MuzzlePoint_R
		```


		하지만 지금 **로봇 본체 블루프린트**를 만들 때 고르는 부모 클래스는 아님.
